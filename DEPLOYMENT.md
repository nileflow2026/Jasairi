## 1. Architecture Overview

```
┌─────────────────────────────────────────────────────┐
│                   JASIRI PLATFORM                    │
├─────────────────────────┬───────────────────────────┤
│   Mobile App (Expo)     │   Backend API (Express)   │
│   EAS Build + Submit    │   Railway.app              │
│   OTA: expo-updates     │   Node 18 LTS              │
├─────────────────────────┼───────────────────────────┤
│         Appwrite Cloud (Auth + DB + Storage)         │
├─────────────────────────┬───────────────────────────┤
│   Redis (Railway Add-on)│   Logs (Railway + Sentry) │
└─────────────────────────┴───────────────────────────┘
```

### Technology Choices

| Layer | Technology | Reason |
|---|---|---|
| Mobile | Expo EAS | Zero-config iOS/Android builds + OTA updates |
| Backend | Railway | Zero-config Node.js, env vars UI, instant rollback |
| Database/Auth | Appwrite Cloud | Already integrated; managed, COPPA-friendly |
| Cache | Upstash Redis | Serverless Redis — no idle cost; Railway add-on as fallback |
| Monitoring | Sentry + Railway metrics | Crash tracking + infra metrics |

## 2. Pre-launch Checklist

Complete every item before tagging `v1.0.0`.

### Backend

- [ ] All `z.ZodError` config fields pass validation with production values
- [ ] `NODE_ENV=production` confirmed
- [ ] JWT secrets are ≥ 32 random characters (use `openssl rand -hex 32`)
- [ ] `BCRYPT_ROUNDS=12` (do not lower)
- [ ] `CORS_ORIGIN` locked to production mobile bundle IDs — no wildcards
- [ ] `/health` endpoint returns `200` with all expected fields
- [ ] Rate limiter tested under load (100 req / 15 min per IP by default)
- [ ] All Appwrite collection indexes created via `npm run setup:db`
- [ ] Redis connection verified in production environment
- [ ] Winston log level set to `info` (not `debug`)
- [ ] `CONTENT_MODERATION_ENABLED=true`
- [ ] No `.env` file committed to git (check `.gitignore`)

### Mobile App

- [ ] `app.json` `version` and `android.versionCode` incremented
- [ ] `EXPO_PUBLIC_API_URL` points to production backend URL
- [ ] EAS project ID registered in `eas.json`
- [ ] Splash screen and app icon assets present in `assets/images/`
- [ ] `expo-secure-store` used for all token storage (not AsyncStorage)
- [ ] Deep link scheme `jasiri://` configured in `app.json`
- [ ] Android adaptive icon backgrounds present (all four variants)
- [ ] Production build tested on a low-end Android device (≤ 2GB RAM)

### Security

- [ ] No hardcoded secrets anywhere in source code
- [ ] Appwrite API key has minimum required permissions only
- [ ] All child data fields comply with COPPA (no PII beyond what guardian provides)
- [ ] `helmet` middleware active with HSTS preload
- [ ] Request IDs (X-Request-Id) present in all responses

## 3. Secrets Management

### Rule: Never commit secrets to git.

All secrets live in:
- **Local dev**: `Backend/.env` (gitignored)
- **CI/CD**: GitHub Actions encrypted secrets
- **Production**: Railway environment variable dashboard
- **Mobile**: Expo EAS secrets + `app.config.js` env injection

### Required Production Secrets

| Variable | How to Generate | Where Used |
|---|---|---|
| `JWT_SECRET` | `openssl rand -hex 32` | Backend |
| `JWT_REFRESH_SECRET` | `openssl rand -hex 32` | Backend |
| `APPWRITE_API_KEY` | Appwrite Console → API Keys | Backend |
| `APPWRITE_PROJECT_ID` | Appwrite Console → Settings | Backend + Mobile |
| `REDIS_URL` | Upstash dashboard / Railway add-on | Backend |
| `REDIS_PASSWORD` | Auto-generated by provider | Backend |
| `SENTRY_DSN` | Sentry project settings | Backend |
| `EXPO_PUBLIC_API_URL` | Your Railway backend URL | Mobile build |

### GitHub Actions Secrets Setup

```
Repository → Settings → Secrets and variables → Actions

Add each secret:
- RAILWAY_TOKEN          (from Railway account settings)
- EXPO_TOKEN             (from expo.dev account settings)
- APPWRITE_API_KEY
- APPWRITE_PROJECT_ID
- JWT_SECRET
- JWT_REFRESH_SECRET
- REDIS_URL
- REDIS_PASSWORD
- SENTRY_DSN
- EXPO_PUBLIC_API_URL
```

### Secret Rotation Policy

| Secret | Rotation Frequency | Trigger |
|---|---|---|
| JWT_SECRET | Every 90 days | Schedule or breach |
| JWT_REFRESH_SECRET | Every 90 days | Schedule or breach |
| APPWRITE_API_KEY | On team member departure | Immediately |
| REDIS_PASSWORD | Every 180 days | Schedule |

After rotating JWT secrets: all active sessions are invalidated — users must re-login.
Coordinate rotations during low-traffic windows (East Africa: 3–5 AM EAT).

## 4. Backend Deployment — Railway

### Why Railway

- Zero-config Node.js detection
- Built-in env variable management
- Instant rollback to any prior deployment
- Automatic TLS certificates
- Built-in Redis add-on
- Region: `us-east` (closest low-latency option to East Africa on Railway free tier; upgrade to a closer region on paid plan)

### Initial Setup

```bash
# 1. Install Railway CLI
npm install -g @railway/cli

# 2. Login
railway login

# 3. Inside Backend/ directory — link to your project
cd Backend
railway init
# Choose: Create a new project → Name: jasiri-backend

# 4. Set all environment variables
railway variables set NODE_ENV=production
railway variables set PORT=3000
railway variables set APPWRITE_ENDPOINT=https://cloud.appwrite.io/v1
railway variables set APPWRITE_PROJECT_ID=<value>
railway variables set APPWRITE_API_KEY=<value>
railway variables set JWT_SECRET=<openssl rand -hex 32>
railway variables set JWT_EXPIRES_IN=7d
railway variables set JWT_REFRESH_SECRET=<openssl rand -hex 32>
railway variables set JWT_REFRESH_EXPIRES_IN=30d
railway variables set REDIS_URL=<from Railway Redis add-on>
railway variables set REDIS_PASSWORD=<from Railway Redis add-on>
railway variables set LOG_LEVEL=info
railway variables set LOG_FILE=logs/app.log
railway variables set BCRYPT_ROUNDS=12
railway variables set MAX_FILE_SIZE=10mb
railway variables set UPLOAD_FOLDER=uploads
railway variables set RATE_LIMIT_WINDOW_MS=900000
railway variables set RATE_LIMIT_MAX_REQUESTS=100
railway variables set CORS_ORIGIN=https://your-expo-domain.com
railway variables set CONTENT_MODERATION_ENABLED=true
railway variables set MAX_CHILDREN_PER_GUARDIAN=5
railway variables set SESSION_TIMEOUT_MINUTES=30

# 5. Deploy
railway up
```

### railway.json (project root of Backend/)

```json
{
  "$schema": "https://railway.app/railway.schema.json",
  "build": {
    "builder": "NIXPACKS"
  },
  "deploy": {
    "startCommand": "node src/server.js",
    "healthcheckPath": "/health",
    "healthcheckTimeout": 30,
    "restartPolicyType": "ON_FAILURE",
    "restartPolicyMaxRetries": 3
  }
}
```

### Nixpacks Build (auto-detected by Railway)

Railway reads `"engines": { "node": ">=18.0.0" }` from `package.json` and pins Node 18 LTS. No `Dockerfile` required unless you need custom build steps.

### Redis Add-on

```
Railway Dashboard → Your Project → + New → Database → Redis
Copy REDIS_URL and REDIS_PASSWORD into environment variables
```

### Custom Domain

```
Railway Dashboard → Your Backend Service → Settings → Networking
Add domain: api.jasiri.app → CNAME to provided Railway URL
TLS is automatic.
```

### Staging Environment

Create a second Railway environment:

```bash
railway environment create staging
railway environment use staging
railway variables set NODE_ENV=staging
# Set staging-specific values — use separate Appwrite project for staging
railway up
```

## 5. Mobile App Deployment — EAS

### Install EAS CLI

```bash
npm install -g eas-cli
eas login
```

### Register the Project

```bash
cd Jasiri
eas init
# This writes your EAS project ID into app.json
```

### app.config.js (create this — replaces static app.json for env injection)

```js
// Jasiri/app.config.js
export default ({ config }) => ({
  ...config,
  extra: {
    apiUrl: process.env.EXPO_PUBLIC_API_URL,
    appwriteProjectId: process.env.EXPO_PUBLIC_APPWRITE_PROJECT_ID,
    appwriteEndpoint: process.env.EXPO_PUBLIC_APPWRITE_ENDPOINT,
    eas: {
      projectId: process.env.EAS_PROJECT_ID,
    },
  },
});
```

### EAS Build Profiles (eas.json)

```json
{
  "cli": {
    "version": ">= 10.0.0"
  },
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal",
      "android": {
        "buildType": "apk"
      }
    },
    "preview": {
      "distribution": "internal",
      "android": {
        "buildType": "apk"
      },
      "env": {
        "EXPO_PUBLIC_API_URL": "https://jasiri-backend-staging.up.railway.app"
      }
    },
    "production": {
      "distribution": "store",
      "android": {
        "buildType": "app-bundle"
      },
      "env": {
        "EXPO_PUBLIC_API_URL": "https://api.jasiri.app"
      }
    }
  },
  "submit": {
    "production": {
      "android": {
        "serviceAccountKeyPath": "./google-play-key.json",
        "track": "internal"
      }
    }
  }
}
```

### Build Commands

```bash
# Internal preview APK (for testers on low-end Android devices)
eas build --profile preview --platform android

# Production Android App Bundle (for Google Play)
eas build --profile production --platform android

# Production iOS (requires Apple Developer account)
eas build --profile production --platform ios

# Submit directly to stores
eas submit --platform android --profile production
eas submit --platform ios --profile production
```

### OTA Updates (expo-updates)

OTA updates push JS bundle changes without going through the app store.
Use for: bug fixes, content updates, UI tweaks.
Do NOT use for: native module changes, new permissions, SDK upgrades.

```bash
# Push an OTA update to all production users
eas update --branch production --message "Fix: memory game scoring"

# Push to preview branch only (for caregiver beta testers)
eas update --branch preview --message "Test: new art studio colors"
```

### EAS Secrets (for CI builds)

```bash
# Store secrets that are injected at build time
eas secret:create --name EXPO_PUBLIC_API_URL --value https://api.jasiri.app --scope project
eas secret:create --name EXPO_PUBLIC_APPWRITE_PROJECT_ID --value <value> --scope project
eas secret:create --name EXPO_PUBLIC_APPWRITE_ENDPOINT --value https://cloud.appwrite.io/v1 --scope project
```

### Google Play Setup

1. Create app in [Google Play Console](https://play.google.com/console) → "Jasiri Learning"
2. Category: Education
3. Content rating: Complete questionnaire (this is a children's app — rated Everyone)
4. Target audience: Children (triggers COPPA compliance requirements in Play Console)
5. Download service account JSON key → save as `Jasiri/google-play-key.json` (gitignored)
6. Start with **Internal Testing** track → promote to **Production** after QA

## 6. Appwrite Cloud Setup

Appwrite is already the primary data store. Use Appwrite Cloud (not self-hosted) for the MVP.

### Project Structure

| Environment | Appwrite Project |
|---|---|
| Development | `jasiri-dev` |
| Staging | `jasiri-staging` |
| Production | `jasiri-prod` |

Keep production isolated — never point staging at the production Appwrite project.

### API Key Permissions (Production)

Create a dedicated API key in Appwrite Console with **only** these scopes:

```
databases.read
databases.write
users.read
users.write
storage.read
storage.write
```

Never use the project's admin key in the backend `.env`.

### Collections Setup

Run the existing setup scripts against each environment:

```bash
# Development
APPWRITE_PROJECT_ID=jasiri-dev node scripts/setup-all-collections.js

# Staging
APPWRITE_PROJECT_ID=jasiri-staging node scripts/setup-all-collections.js

# Production (run once, carefully)
APPWRITE_PROJECT_ID=jasiri-prod node scripts/setup-all-collections.js
```

### Backup Policy

- Appwrite Cloud automatically retains 7 days of database backups on paid plans
- Export critical collections weekly using Appwrite's export API or the Console
- Store exports in an encrypted S3-compatible bucket (Cloudflare R2 is free tier)

## 7. Redis in Production

Redis backs the rate limiter and any session caching. Two options:

### Option A: Railway Redis Add-on (Recommended for MVP)

- Provisioned alongside the backend service in Railway
- No separate billing until you exceed the free tier
- `REDIS_URL` is auto-injected as an environment variable

### Option B: Upstash (Serverless — zero idle cost)

```
upstash.com → Create Database → Region: af-south (Cape Town) or eu-west
Copy REST URL and token → set as REDIS_URL and REDIS_PASSWORD
```

Upstash is preferred when traffic is unpredictable — you pay per request not per uptime.

### Redis Security

- Always set `REDIS_PASSWORD` — never run Redis without authentication
- Use TLS (`rediss://` not `redis://`) in production
- Restrict Redis to only accept connections from the backend service IP

## 8. CI/CD Pipelines

Both workflows live in `.github/workflows/`.

### Backend CD — `.github/workflows/backend-deploy.yml`

## 9. Monitoring & Health Checks

### Health Endpoint

The backend already exposes `GET /health`. Use it for:

```bash
# Manual check
curl https://api.jasiri.app/health

# Expected response
{
  "status": "healthy",
  "version": "1.0.0",
  "environment": "production",
  "uptime": 3600,
  "timestamp": "2026-05-18T...",
  "system": { ... }
}
```

### Railway Health Check

Already configured in `railway.json` → `healthcheckPath: /health`.
Railway will restart the service if `/health` returns non-200 for 30 seconds.

### Sentry Integration (Backend)

```bash
npm install @sentry/node --save
```

Add to `src/app.js` before all other middleware:

```js
const Sentry = require('@sentry/node');

Sentry.init({
  dsn: process.env.SENTRY_DSN,
  environment: process.env.NODE_ENV,
  tracesSampleRate: process.env.NODE_ENV === 'production' ? 0.1 : 1.0,
});
```

Add Sentry error handler after your routes:

```js
app.use(Sentry.Handlers.errorHandler());
```

### Alerting Thresholds

| Metric | Warning | Critical | Action |
|---|---|---|---|
| API response time | > 2s | > 5s | Check Railway metrics, Redis latency |
| Error rate | > 1% | > 5% | Check Sentry, review logs |
| Heap used | > 400 MB | > 600 MB | Restart service, investigate memory leak |
| Redis latency | > 100ms | > 500ms | Check Upstash region, connection pool |
| /health non-200 | 1 occurrence | 3 consecutive | Railway auto-restarts; page on-call |

## 10. Rollback Strategies

### Backend Rollback (Railway)

Railway keeps full deployment history. To roll back:

**Via Dashboard:**
```
Railway Dashboard → Your Project → Deployments tab
→ Click any prior deployment → "Redeploy"
```

**Via CLI:**
```bash
# List recent deployments
railway deployments list

# Roll back to a specific deployment ID
railway deployments redeploy <deployment-id>
```

Rollback completes in under 60 seconds (Railway swaps the running container).

**Environment variable rollback:**
If a bad secret was deployed, fix the variable first, then redeploy:
```bash
railway variables set JWT_SECRET=<corrected-value>
railway up
```

### Mobile App Rollback

#### OTA Rollback (JS-only changes — fastest)

```bash
# List update branches
eas update:list --branch production

# Roll back by republishing the previous update
eas update --branch production --message "Rollback to <previous-id>"
```

OTA rollback reaches all users within minutes (on next app launch).

#### Store Build Rollback

If a bad build was promoted to the Play Store production track:

```
Google Play Console → Release → Production
→ "Create new release" → upload the last known-good .aab file
→ Rollout to 100%
```

Keep the last 3 production `.aab` files archived locally and in your CI artifacts.

#### OTA Disable (Emergency)

If an OTA update causes a crash loop:

```bash
# Point production channel back to the last store build (no OTA)
eas channel:edit production --branch <last-stable-branch>
```

This forces all users to use the binary they installed from the store, bypassing the broken OTA update.

### Database Rollback (Appwrite)

Appwrite Cloud does not support point-in-time restore on the free plan.

Mitigation strategy:
1. Before any schema change, export affected collections from the Appwrite Console
2. Store the export in `/backups/appwrite/YYYY-MM-DD/`
3. To restore: re-import via Appwrite Console or the Node.js SDK

For collection schema changes, always add new fields as **optional** first before removing old ones.

## 11. Incident Response

### Severity Levels

| Level | Definition | Response Time |
|---|---|---|
| P0 | App completely unusable for all users | Immediate (< 15 min) |
| P1 | Core feature broken (auth, games) | < 1 hour |
| P2 | Non-critical feature degraded | < 4 hours |
| P3 | Minor visual/UX issue | Next sprint |

### P0 Response Playbook

```
1. Confirm incident via Sentry + Railway health check
2. Identify: backend, OTA update, or Appwrite outage?
3. Backend issue → Railway rollback (< 60 sec)
4. OTA issue    → eas channel:edit to stable branch
5. Appwrite down → check status.appwrite.io, no action needed (managed)
6. Notify users via in-app banner (add to next OTA push)
7. Post-mortem within 48 hours
```

### Status Page

Recommended: [Instatus](https://instatus.com) (free tier) for a public status page.
Subscribe your status page to the Railway and Appwrite status RSS feeds for automatic propagation.
